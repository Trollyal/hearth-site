the shift

You stopped writing code.
You started directing it.

The old loop was one session guessing its way through a task. v1 splits the roles: your session is the PM that composes and gates; a deterministic executor does the building. No more reading stdout to guess what happened.

before — one session, one thread

Prompt, hope, re-prompt

  • One agent juggles building, testing, and remembering — context bloats, then resets.
  • Progress is invisible — you scroll a transcript to find out what changed.
  • "Did it verify?" is a vibe, not a verdict. Findings get lost between turns.
v1 — orchestrator + executor

Compose, fire, gate

  • Your session composes the work as a typed graph and hands it off — then stays free to talk.
  • The executor walks it deterministically: build → verify loops, carrying verifier findings into the next build.
  • Every node lights up live on a canvas. PASS commits and opens a delivery gate — automatically.
design the work

Flows, not prompts.

Compose a graph from typed nodes — build, verify, research, synthesize, human-gate — wired by typed edges. Author it on the canvas, or describe the task in a sentence and let the planner draft it. The executor handles the rest: bounded loops, verdict-gated branches, parallel fan-in.

Build ⇄ Verify loop

Verify fails? The executor loops back to build with the findings attached, up to the node's iteration budget. PASS commits.

Parallel fan-in

Run a panel of read-only assessors at once — security, architecture, UX — and fold their findings into one synthesis node.

Verdict branch & gate

Route on the verifier's verdict. PASS heads to a human gate for sign-off; FAIL composes a scoped fix flow. You hold the checkpoints that matter.

The executor shells one agent per node under an enforced output contract — no stdout-sniffing. Every read & write is attributed to its repo; a write to a read-only mount is flagged loudly.

the control plane

Watch the whole forge at a glance.

A desktop app over the same file-based state — no IPC, no server. The run canvas shows the live graph; the feed rail beside it carries events, decisions, and the gates waiting on you. Approve, request changes, or take over the terminal, all in one place.

what's inside

Built for real, repo-aware work.

Flows & the run canvas

Author the graph the executor runs, or let the planner draft it. Watch runs light up node by node, with full read/write provenance on every touch.

Topics & mounts

A topic binds one repo, several, or none — each read-only or read-write. Isolated worktrees, branch, and state. No crossed wires.

Crews & personas

Builders, verifiers, architects — specialist agents with scoped tools, staffed per node and held to acceptance criteria.

Delivery gates

When a run passes, the app commits the work and opens a delivery gate with the diff attached — so nothing leaves the topic without your go-ahead.

Repo adapters

Drop a .claude/hearth/ overlay in any repo — MR conventions, dev-server, integrations, skills. Or let an agent draft it for you.

Scheduled & autonomous

An in-app scheduler pulls sources and runs triage on a cadence — the queue stays warm whether the app is open or closed.

one state, three surfaces

App, CLI, and MCP — over the same files.

Every event, command, decision, and run record is a file on disk. The work survives context resets, and every surface reads the same state — no database, no IPC, no drift.

Desktop app

The control plane — run canvas, feed rail, inbox of decisions, adapters. Oversight at a glance.

CLI

One binary for speed — status, dispatch, tail crews, manage topics. The forge from the terminal.

MCP server

Exposes the whole state to any Claude Code session — plan flows, fire runs, raise gates, from your editor.
