HEARTH

Your coding session becomes a project manager.

You compose the work as a flow — build, verify, audit, gate — and Hearth's executor runs it for you, step by step, on a live canvas. All coordinated through durable file-based state, with human gates where they matter.

the idea

One session orchestrates. The app runs the work.

The main thread never writes code. It reads the work, composes it as a flow, fires it, and surfaces a decision only when one is genuinely needed.

01

The session is the PM

Your main thread composes the work as a flow and fires it, then supervises and gates the result. It coordinates — it never writes code.

02

The executor runs the loop

Build → verify, looping on FAIL with findings carried forward; parallel assessors fanning into a synthesis; branches and gates. Deterministic orchestration, not a prompt guessing what happened.

03

State lives on disk

Every event, command, decision, and run record is a file. The work survives context resets — and the companion app reads the same files, no IPC.

04

You hold the gates

Ambiguous calls become explicit human checkpoints. Everything else runs autonomously until there's a real decision to make.

Design the work as a flow.

what's inside

Built for real, repo-aware work.

Flows & the run canvas

Author the graph the executor runs — build/verify loops, audits, gates — on a canvas, or let the planner draft it. Watch runs light up node by node, with full read/write provenance.

Topics & mounts

A topic binds one or more repos — each read-only or read-write — or none at all for research and automation. Isolated worktrees, branch, and state. No crossed wires.

Crews & personas

Builders, reviewers, verifiers, architects — specialist agents with scoped tools, dispatched in parallel and held to acceptance criteria.

Repo adapters

Drop a .claude/hearth/ overlay in any repo — MR conventions, dev-server, integrations, skills. Or let an agent draft it for you.

Intake & triage

Optional sources (like Notion) feed a backlog. Each ticket is triaged — complexity, suggested crew, risk flags — for you to accept.

The feed rail

Events, decisions, gates, and chat on one timeline — the conversation rail beside the live run canvas. Take over the terminal, request changes, sign off — all in one place.

App · CLI · MCP

Same file-based state, three surfaces. The desktop app for oversight, the CLI for speed, and an MCP server exposing it all to any Claude Code session.

Scheduled & autonomous

An in-app scheduler pulls sources and runs triage on a cadence — the queue stays warm whether the app is open or closed.

AI-drafted adapters

Point the app at a repo — an agent reads its CI, scripts, and branch conventions and drafts the .claude/hearth/ overlay for you to approve.

The Master Smith

A system-aware PM that knows Hearth itself — framework, app, and CLI. Your turn-to when you're improving the forge, not just working in it.
